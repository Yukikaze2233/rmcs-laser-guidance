# 后续 RMCS 接入方向

这份文档只描述**下一阶段可能怎么接**，不代表当前就要实现。

## 预期接入顺序

如果将来要接入 RMCS，建议按下面顺序加：

1. 增加 bridge 组件
2. 明确视觉结果如何映射成控制所需信息
3. 再决定是否接 `/tf`
4. 再决定是否输出 `/gimbal/laser_guidance/*`

不要反过来先定义一堆内部总线接口，再去补视觉实现。

## 推荐的 bridge 职责

bridge 只做三类事：

- 参数读取
- 输入采集
- 结果发布

bridge 不应该成为算法实现本身。

## 可能的输入

第二阶段可能会消费：

- `/tf`
- `/predefined/timestamp`
- 未来若需要同步曝光，再考虑 `HardSyncSnapshot`

但当前阶段不需要。

## 可能的输出

如果未来需要接控制，建议先定义独立命名空间：

```text
/gimbal/laser_guidance/...
```

而不是直接复用：

```text
/gimbal/auto_aim/...
```

这样不会和 `sp_vision_25` 的接口混淆。

## 为什么现在不做

因为当前最小骨架的风险不在控制，而在视觉链本身：

- 相机是否稳定
- 检测接口是否足够干净
- 调试路径是否足够短

这些问题没解决前，接 RMCS 只会把调试复杂度迅速放大。
